# Writing Tasks

Tasks are step-by-step workflows that agents execute. Learn how to write clear, actionable tasks with elicitation patterns and validation gates.

## What is a Task?

A task is a **deterministic workflow** that defines:
- **Inputs** - What data the task needs
- **Steps** - Sequential actions to perform
- **Outputs** - What the task produces
- **Validation** - How to verify completion
- **Error Handling** - What to do when things fail

Tasks eliminate improvisation and ensure consistent execution across different contexts.

## Task File Structure

```markdown
# Task: Task Title

**Task ID:** task-identifier
**Version:** 1.0
**Purpose:** Clear one-line description
**Orchestrator:** @agent-id
**Mode:** interactive | autonomous

## Overview

Detailed description of what this task accomplishes and when to use it.

## Inputs

| Parameter | Type | Required | Description | Example |
|-----------|------|----------|-------------|----------|
| param1 | string | Yes | Description | "value" |
| param2 | number | No | Description | 42 |

## Preconditions

- [ ] Condition 1 must be met
- [ ] Condition 2 must be met

## Steps

### Step 1: Step Name

**Actions:**
- Action 1
- Action 2

**Output:** What this step produces

### Step 2: Next Step

...

## Validation Criteria

- [ ] Output exists and is valid
- [ ] No errors logged
- [ ] Quality gates passed

## Error Handling

**Common Errors:**
1. **Error Name** - Cause and resolution

## Outputs

| Output | Location | Description |
|--------|----------|-------------|
| file | path/to/file | What it contains |
```

## Creating a Task

<Steps>

### Step 1: Define Task Purpose

Start with a clear, specific purpose:

```markdown
# Task: Generate API Documentation

**Task ID:** generate-api-docs
**Version:** 1.0
**Purpose:** Generate comprehensive API documentation from OpenAPI spec
**Orchestrator:** @dev
**Mode:** autonomous
```

### Step 2: Specify Inputs

Define all required and optional parameters:

```markdown
## Inputs

| Parameter | Type | Required | Description | Example |
|-----------|------|----------|-------------|---------|
| spec_file | string | Yes | Path to OpenAPI spec | "api/openapi.yaml" |
| output_dir | string | No | Output directory | "docs/api" |
| format | string | No | Output format (html\|markdown) | "markdown" |
| include_examples | boolean | No | Include code examples | true |
```

### Step 3: Add Preconditions

Validate requirements before execution:

```markdown
## Preconditions

- [ ] OpenAPI spec file exists and is valid YAML/JSON
- [ ] Output directory is writable
- [ ] Required tools installed (swagger-cli, redoc-cli)
```

### Step 4: Write Step-by-Step Actions

Break the task into sequential steps:

```markdown
## Steps

### Step 1: Validate OpenAPI Spec

**Actions:**
```bash
# Validate spec syntax
swagger-cli validate {spec_file}
```

**Validation:**
- Spec must be valid OpenAPI 3.0+ format
- All references must resolve
- No schema errors

**Output:** Validation report

### Step 2: Generate Documentation

**Actions:**
```bash
# Generate markdown docs
redoc-cli bundle {spec_file} -o {output_dir}/index.html

# Generate markdown for each endpoint
node .aiox-core/scripts/generate-endpoint-docs.js {spec_file} {output_dir}
```

**Output:** 
- HTML documentation at `{output_dir}/index.html`
- Markdown files for each endpoint

### Step 3: Add Code Examples

**Condition:** Only if `include_examples == true`

**Actions:**
- Extract example requests/responses from spec
- Generate code examples in multiple languages
- Add to documentation pages

**Output:** Code example files

### Step 4: Validate Generated Docs

**Actions:**
- Check all endpoint files created
- Verify links are valid
- Validate HTML structure
- Run accessibility checks

**Validation:**
```bash
# Check for broken links
npm run validate:docs {output_dir}
```
```

### Step 5: Define Validation Criteria

Specify completion requirements:

```markdown
## Validation Criteria

- [ ] All API endpoints documented
- [ ] Code examples included for each endpoint
- [ ] No broken links in documentation
- [ ] HTML validates with no errors
- [ ] Documentation accessible (WCAG 2.1 AA)
- [ ] Navigation structure is correct
```

### Step 6: Add Error Handling

Document common failures and resolutions:

```markdown
## Error Handling

**Common Errors:**

1. **Invalid OpenAPI Spec**
   - **Cause:** YAML syntax errors or invalid schema
   - **Resolution:** Run `swagger-cli validate` and fix reported errors
   - **Recovery:** Fix spec file, retry from Step 1

2. **Missing Dependencies**
   - **Cause:** Required tools not installed
   - **Resolution:** Run `npm install -g swagger-cli redoc-cli`
   - **Recovery:** Install tools, retry from Step 1

3. **Permission Denied**
   - **Cause:** Cannot write to output directory
   - **Resolution:** Check directory permissions, create if missing
   - **Recovery:** Fix permissions, retry from Step 2
```

</Steps>

## Elicitation Patterns

Elicitation is the process of gathering information from users. Use it for interactive tasks.

### When to Use Elicitation

Use `elicit: true` when:
- Task needs user decisions
- Multiple valid approaches exist
- Configuration requires user preferences
- Ambiguity needs clarification

### Elicitation Structure

```markdown
## Interactive Elicitation Process

### Phase 1: Gather Requirements

```
ELICIT: API Documentation Requirements

1. What endpoints should be documented?
   Options:
   a) All endpoints (recommended)
   b) Public API only
   c) Specific endpoints (list below)

2. What information should be included?
   [ ] Request/response examples
   [ ] Authentication details
   [ ] Rate limiting info
   [ ] Error codes and meanings
   [ ] Code examples (multiple languages)

3. What output format?
   a) HTML (interactive, searchable)
   b) Markdown (simple, Git-friendly)
   c) Both

4. Should we generate API client libraries?
   [ ] Yes - specify languages: __________
   [ ] No
```

**Validation:**
- Verify at least one endpoint selected
- Confirm output format chosen
- Validate language selection if clients requested

### Phase 2: Confirm Configuration

```
CONFIRM: Review API Documentation Plan

Endpoints: \{count\} endpoints
Format: \{format\}
Examples: \{yes/no\}
Client Libraries: \{languages or "none"\}

Proceed with these settings? (yes/no)
```
```

### Elicitation Best Practices

<CardGroup cols={2}>

<Card title="Numbered Options" icon="list-ol">
Provide numbered choices so users can type "1" instead of full text.
</Card>

<Card title="Clear Defaults" icon="star">
Indicate recommended options with "(recommended)" or "(default)".
</Card>

<Card title="Validation" icon="shield-check">
Validate user input immediately and provide helpful error messages.
</Card>

<Card title="Confirmation" icon="check-double">
Summarize choices before execution and ask for final confirmation.
</Card>

## Gate Implementation

Gates are validation checkpoints that block execution if conditions aren't met.

### Quality Gates

```markdown
## Quality Gate: API Documentation

**Gate ID:** QG_API_DOCS_001
**Blocking:** Yes
**Trigger:** After Step 4 (before marking complete)

### Blocking Requirements

```yaml
blocking_requirements:
  endpoint_coverage: 100%  # All endpoints documented
  broken_links: 0          # No broken links
  examples_present: true   # Examples for all endpoints
  html_valid: true         # Valid HTML5
```

### Scoring

| Dimension | Weight | Check |
|-----------|--------|-------|
| Coverage | 0.30 | All endpoints documented |
| Examples | 0.25 | Code examples present |
| Links | 0.20 | No broken links |
| Validation | 0.15 | HTML/Markdown valid |
| Accessibility | 0.10 | WCAG 2.1 AA |

**Threshold:** 8.0/10

### Veto Conditions

```yaml
veto_conditions:
  - endpoint_coverage < 80% → "Insufficient coverage"
  - broken_links > 5 → "Too many broken links"
  - examples_present == false → "Missing code examples"
```

**Decision Point:**

```text
IF score >= 8.0 AND no veto conditions:
    → PASS - Documentation ready
ELSE IF score >= 6.0 AND veto_conditions resolved:
    → CONDITIONAL PASS - Document gaps
ELSE:
    → FAIL - Fix issues and retry
```
```

### Checkpoint Pattern

```markdown
### Checkpoint: Validate Generated Files

**Checkpoint ID:** CP_FILES_EXIST
**Blocking:** Yes

**Validations:**
```bash
# Check all required files exist
test -f {output_dir}/index.html
test -d {output_dir}/endpoints
test -f {output_dir}/endpoints/get-users.md
```

**On Failure:**
- Log missing files
- Return to Step 2 (regenerate)
- Max retries: 2
- If still failing: HALT and notify user
```

## Task Naming Conventions

**Agent-specific tasks:**
- Format: `{agent-id}-{action}.md`
- Example: `dev-develop-story.md`, `qa-review-code.md`
- Used by: Single agent only

**Shared tasks:**
- Format: `{action}.md` (no prefix)
- Example: `execute-checklist.md`, `create-doc.md`
- Used by: Multiple agents

**Naming guidelines:**
- Use kebab-case
- Use verb-noun pattern (`create-agent`, `analyze-performance`)
- Be specific and descriptive
- Avoid generic names (`process`, `handle`, `do`)

## Best Practices

<AccordionGroup>

<Accordion title="Write for Machines">
**Tasks are executed by LLMs, not humans.**

- Be explicit - don't assume context
- Use structured formats (YAML, JSON)
- Provide exact commands with placeholders
- Define success criteria precisely
- Include validation commands
</Accordion>

<Accordion title="Make Steps Atomic">
**Each step should be independently executable.**

- One clear objective per step
- Explicit inputs and outputs
- No hidden dependencies
- Can resume from any step
- Step failure doesn't break entire task
</Accordion>

<Accordion title="Handle Errors Gracefully">
**Anticipate failure modes.**

- Document common errors
- Provide recovery steps
- Set max retry limits
- Define escalation points
- Log failures for debugging
</Accordion>

<Accordion title="Use Veto Conditions">
**Block execution when requirements aren't met.**

- Define blocking conditions
- Explain why each blocks
- Provide fix guidance
- Set quality thresholds
- Prevent incomplete work
</Accordion>

<Accordion title="Include Examples">
**Show expected inputs and outputs.**

- Provide real-world examples
- Show edge cases
- Include error scenarios
- Demonstrate validation
- Link to actual output
</Accordion>

</AccordionGroup>
