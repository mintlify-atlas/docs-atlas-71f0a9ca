# ⚡ Gage - DevOps Specialist

**Role**: GitHub Repository Guardian & Release Manager  
**Archetype**: Operator ♈ Aries  
**Persona**: Systematic, quality-focused, security-conscious, detail-oriented

## When to Use Gage

Activate `@devops` for:

- 🚀 **Git push to remote** (EXCLUSIVE AUTHORITY)
- 📦 Pull request creation and management
- 🏷️ Semantic versioning and release management
- ⚙️ CI/CD pipeline configuration (GitHub Actions)
- 🧹 Repository cleanup (stale branches)
- 👥 GitHub Issues triage and resolution
- 🔧 Environment health diagnostics

**NOT for**: Code implementation (use `@dev`), Story creation (use `@sm`), Architecture design (use `@architect`)

<Warning>
**CRITICAL**: Gage is the **ONLY** agent authorized to execute `git push` to remote repository.

All other agents must delegate push operations to DevOps.
</Warning>

## Git Push Workflow

### Standard Push Process

```bash
# 1. Story is complete and QA approved
# Status: "Ready for Review"

# 2. Activate Gage
@devops

# 3. Run pre-push quality gates
*pre-push

# Gage will run:
# - CodeRabbit review (--base main)
# - npm run lint
# - npm test
# - npm run typecheck
# - npm run build
# - Story status check
# - Uncommitted changes check
# - Merge conflict check

# 4. If ALL PASS, Gage presents summary
# 5. User confirms
# 6. Gage executes: git push

# 7. If on feature branch, Gage offers:
*create-pr
```

### Quality Gate Summary

```yaml
Pre-Push Quality Gates:

  ✅ CodeRabbit Review:
     - CRITICAL issues: 0
     - HIGH issues: 0
     - MEDIUM issues: 2
  
  ✅ Linting: PASS
  ✅ Type Checking: PASS
  ✅ Tests: PASS (47/47)
  ✅ Build: PASS
  ✅ Story Status: "Ready for Review"
  ✅ No uncommitted changes
  ✅ No merge conflicts

All checks passed! Ready to push?
[y/N]
```

### CodeRabbit Pre-PR Gate

```bash
*create-pr

# Gage will:
# 1. Run CodeRabbit against main branch
# 2. Check for CRITICAL issues
# 3. If CRITICAL found: BLOCK PR creation
# 4. If HIGH found: WARN user, allow override
# 5. If clean or approved: Create PR
```

**Severity handling**:

<Tabs>
  <Tab title="CRITICAL">
    **Action**: Block PR creation, must fix immediately
    
    **Examples**:
    - Security vulnerabilities
    - Breaking changes without migration
    - Data loss risks
    
    **Response**:
    ```
    ❌ Cannot create PR - CRITICAL issues found:
    
    1. SQL injection risk in api/src/routes/users.js:45
    2. Hardcoded API key in config.js:12
    
    Fix these issues and run *pre-push again.
    ```
  </Tab>
  
  <Tab title="HIGH">
    **Action**: Warn user, recommend fix before merge
    
    **Examples**:
    - Performance issues
    - Missing tests
    - Accessibility violations
    
    **Response**:
    ```
    ⚠️ HIGH priority issues found:
    
    1. Missing index on users.email (performance impact)
    2. Component missing accessibility labels
    
    Proceed with PR creation? (not recommended)
    [y/N]
    ```
  </Tab>
  
  <Tab title="MEDIUM/LOW">
    **Action**: Document in PR description, create follow-up
    
    **Response**:
    ```
    ✅ Quality checks passed!
    
    Note: 2 MEDIUM issues documented in PR description.
    Creating PR...
    ```
  </Tab>
</Tabs>

## Exclusive Git Authority

<Info>
**Enforcement Mechanism**:

Git pre-push hook installed at `.git/hooks/pre-push`:
- Checks `$AIOX_ACTIVE_AGENT` environment variable
- Blocks push if agent != "github-devops"
- Displays helpful message redirecting to @devops
- Works in ANY repository using AIOX
</Info>

**Allowed operations** (DevOps only):
```bash
git push
git push --force         # With extreme caution
git push origin --delete # Branch cleanup
gh pr create
gh pr merge
gh release create
```

**Standard operations** (all agents):
```bash
git status
git log
git diff
git tag
git branch -a
```

## Semantic Versioning

### Version Check

```bash
*version-check

# Gage analyzes:
# - Last tag (e.g., v4.31.0)
# - Git diff since last tag
# - Breaking change keywords
# - Feature additions
# - Bug fixes only

# Recommends:
Current version: v4.31.0
Changes since last tag:
  - 3 features added
  - 7 bug fixes
  - 0 breaking changes

Recommended: v4.32.0 (MINOR bump)
Reason: New features added, backward compatible

Confirm version bump to v4.32.0?
[y/N]
```

### Versioning Rules

<CardGroup cols={3}>
  <Card title="MAJOR" icon="triangle-exclamation" color="#ef4444">
    **Breaking changes**
    
    Examples:
    - API redesign
    - Removed features
    - Database migration required
    
    `v4.31.0` → `v5.0.0`
  </Card>
  
  <Card title="MINOR" icon="plus" color="#10b981">
    **New features**
    
    Examples:
    - New API endpoints
    - New components
    - Backward compatible
    
    `v4.31.0` → `v4.32.0`
  </Card>
  
  <Card title="PATCH" icon="wrench" color="#3b82f6">
    **Bug fixes only**
    
    Examples:
    - Bug fixes
    - Performance improvements
    - Documentation updates
    
    `v4.31.0` → `v4.31.1`
  </Card>
</CardGroup>

## Release Management

```bash
*release

# Gage will:
# 1. Run version-check
# 2. User confirms version
# 3. Run pre-push quality gates
# 4. Generate changelog from commits

# 6. Push tag to remote
```

```bash
*resolve-issue 145

# Gage orchestrates:
# 1. Fetch issue details
# 2. Create feature branch
# 3. Delegate to @dev for implementation
# 4. After @dev completes, run pre-push
# 5. Push and create PR
# 6. Link PR to issue
# 7. Close issue when PR merged
```

## Worktree Management

### Create Isolated Environment

```bash
*create-worktree story-1.2.3

# Gage will:
# 1. Create worktree at .worktrees/story-1.2.3/
# 2. Create feature branch: feature/1.2-story-name
# 3. Set up isolated environment
# 4. Track in worktree registry
```

**Benefits**:
- Work on multiple stories simultaneously
- Clean separation of changes
- Easy rollback per story
- No branch switching conflicts

### Merge and Cleanup

```bash
# Merge worktree back to main
*merge-worktree story-1.2.3

# List active worktrees
*list-worktrees

# Remove stale worktrees (>30 days)
*cleanup-worktrees
```
