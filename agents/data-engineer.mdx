## Migration Workflow

### Safe Migration Process

```bash
# 1. Create snapshot before changes
*snapshot baseline-before-users-table

# 2. Dry run to test migration
*dry-run supabase/migrations/20260305_users.sql

# Dara will:
# - Parse SQL for syntax errors
# - Check DDL ordering (dependencies first)
# - Verify idempotency (IF NOT EXISTS)
# - Validate constraint definitions
# - Report potential issues

# 3. Apply migration if dry run passes
*apply-migration supabase/migrations/20260305_users.sql

# Dara will:
# - Create automatic snapshot
# - Run migration in transaction
# - Validate schema after changes
# - Log migration completion
# - Update migration tracking
```

### Migration Best Practices

<Warning>
**CRITICAL Migration Rules**:

- ✅ Always create snapshot before migration
- ✅ Use IF NOT EXISTS / IF EXISTS for idempotency
- ✅ Order DDL statements (tables before foreign keys)
- ✅ Create rollback script for every migration
- ✅ Test on staging before production
- ✅ Use transactions for multi-statement changes

**NEVER**:

- ❌ Run migrations without dry-run first
- ❌ Skip rollback script creation
- ❌ Use DROP without safeguards
- ❌ Hardcode credentials in migrations
</Warning>

### Rollback Strategy

```bash
# If migration fails or causes issues:

# Option 1: Rollback to snapshot
*rollback baseline-before-users-table

# Option 2: Run rollback script
*rollback supabase/migrations/20260305_users_rollback.sql

# Dara will:
# - Restore database to previous state
# - Verify rollback completed
# - Log rollback action
# - Update migration tracking
```

### RLS Policy Design

```bash
*create-rls-policies

# Dara will:
# 1. Analyze access patterns
# 2. Design policies per table
# 3. Choose KISS or granular approach
# 4. Document policy logic
# 5. Generate rls-policies-tmpl.yaml
```

**RLS Policy Modes**:

<Tabs>
  <Tab title="KISS (Simple)">
    **When to use**: Most tables, straightforward access control
    
    ```sql
    -- Single policy per operation
    CREATE POLICY "Users can view own data"
      ON users FOR SELECT
      USING (auth.uid() = id);
    
    CREATE POLICY "Users can update own data"
      ON users FOR UPDATE
      USING (auth.uid() = id);
    ```
    
    **Benefits**:
    - Easy to understand
    - Simple to maintain
    - Good performance
  </Tab>
  
  <Tab title="Granular (Complex)">
    **When to use**: Multi-tenant, role-based access, complex rules
    
    ```sql
    -- Separate policies for different roles
    CREATE POLICY "Admins can view all users"
      ON users FOR SELECT
      USING (
        EXISTS (
          SELECT 1 FROM user_roles
          WHERE user_id = auth.uid()
          AND role = 'admin'
        )
      );
    
    CREATE POLICY "Users can view own data"
      ON users FOR SELECT
      USING (auth.uid() = id);
    ```
    
    **Benefits**:
    - Fine-grained control
    - Role-based access
    - Multi-tenant support
  </Tab>
</Tabs>

### Index Design

```bash
*design-indexes

# Dara will:
# 1. Analyze query patterns
# 2. Identify columns for indexing
# 3. Choose index types (B-tree, GIN, GIST)
# 4. Design composite indexes
# 5. Consider index maintenance cost
# 6. Generate index-strategy-tmpl.yaml
```

**Indexing principles**:

- ✅ Index foreign keys
- ✅ Index columns in WHERE clauses
- ✅ Index columns in JOIN conditions
- ✅ Index columns in ORDER BY
- ✅ Use partial indexes for filtered queries
- ❌ Don't over-index (slows INSERT/UPDATE)
- ❌ Avoid indexing low-cardinality columns

## CodeRabbit Integration

**Focus**: SQL quality, schema design, query performance, RLS security, migration safety

```bash
# Before applying migration
wsl bash -c 'cd ${PROJECT_ROOT} && ~/.local/bin/coderabbit --prompt-only -t uncommitted'
```

**Severity handling**:

<Tabs>
  <Tab title="CRITICAL">
    **Action**: Block migration/deployment
    
    **Examples**:
    - SQL injection vulnerabilities (string concatenation)
    - Missing RLS policies on public tables
    - Hardcoded credentials in migration scripts
    - DROP statements without safeguards
    - Unsafe SECURITY DEFINER functions
    - Exposure of sensitive data (passwords, tokens, PII)
  </Tab>
  
  <Tab title="HIGH">
    **Action**: Fix before migration or create rollback plan
    
    **Examples**:
    - N+1 query patterns in API code
    - Missing indexes on foreign keys
    - Queries without WHERE clauses on large tables
    - Missing NOT NULL constraints
    - Cascading deletes without safeguards
    - Unoptimized JOIN patterns
    - Memory-intensive queries
  </Tab>
  
  <Tab title="MEDIUM">
    **Action**: Document as technical debt
    
    **Examples**:
    - Denormalization without justification
    - Missing foreign key relationships
    - Lack of comments on complex tables/functions
    - Inconsistent naming conventions
    - Missing audit fields (created_at, updated_at)
    - Unused indexes
  </Tab>
</Tabs>
