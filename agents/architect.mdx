### Architecture Components

<Tabs>
  <Tab title="Technology Stack">
    **Covers**:
    - Frontend framework (React, Vue, Svelte)
    - Backend framework (Express, Fastify, tRPC)
    - Database selection (PostgreSQL, MongoDB, etc.)
    - State management (Zustand, Redux, Jotai)
    - Styling approach (Tailwind, CSS-in-JS)
    - Build tools (Vite, Webpack, Turbopack)
    
    **Selection criteria**:
    - Team experience
    - Project requirements
    - Performance needs
    - Ecosystem maturity
    - "Boring technology" where possible
  </Tab>
  
  <Tab title="API Design">
    **Defines**:
    - API style (REST, GraphQL, tRPC, hybrid)
    - Authentication (JWT, sessions, OAuth)
    - Authorization patterns
    - Rate limiting strategy
    - Error handling conventions
    - Pagination approach
    - Versioning strategy
    
    **Patterns**:
    - RESTful resource design
    - GraphQL schema stitching
    - tRPC procedure organization
    - WebSocket event patterns
  </Tab>
  
  <Tab title="Security Architecture">
    **Layers**:
    1. **Authentication**: JWT, sessions, OAuth providers
    2. **Authorization**: RBAC, ABAC, RLS policies
    3. **Encryption**: TLS, at-rest encryption
    4. **Input Validation**: Schema validation, sanitization
    5. **Rate Limiting**: Per-user, per-IP
    6. **CORS**: Cross-origin policies
    7. **CSP**: Content Security Policy
    8. **Secrets**: Environment variables, vaults
  </Tab>
  
  <Tab title="Infrastructure">
    **Plans**:
    - Hosting platform (Vercel, Railway, AWS)
    - CDN strategy (Cloudflare, Fastly)
    - Database hosting (Supabase, PlanetScale)
    - Caching layer (Redis, Memcached)
    - Monitoring (Sentry, Datadog)
    - Logging (Logtail, CloudWatch)
    - CI/CD pipeline (GitHub Actions)
    - Scaling approach (horizontal, vertical)
  </Tab>
</Tabs>

## CodeRabbit Architectural Review

**Focus**: Architectural patterns, security, anti-patterns, cross-stack consistency

```bash
# Review architectural changes
wsl bash -c 'cd ${PROJECT_ROOT} && ~/.local/bin/coderabbit --prompt-only -t uncommitted'
```

**Severity handling**:

<Tabs>
  <Tab title="CRITICAL">
    **Action**: Block architecture approval
    
    **Focus**:
    - Security vulnerabilities
    - Data integrity risks
    - Critical anti-patterns
    
    **Examples**:
    - Hardcoded credentials
    - SQL injection vulnerabilities
    - Insecure authentication patterns
    - Data exposure risks
  </Tab>
  
  <Tab title="HIGH">
    **Action**: Flag for immediate architectural discussion
    
    **Focus**:
    - Performance bottlenecks
    - Scalability issues
    - Major anti-patterns
    
    **Examples**:
    - N+1 query patterns
    - Missing indexes on critical queries
    - Memory leaks
    - Unoptimized API calls
    - Tight coupling between layers
  </Tab>
  
  <Tab title="MEDIUM">
    **Action**: Document as technical debt
    
    **Focus**:
    - Code maintainability
    - Design patterns
    - Developer experience
    
    **Examples**:
    - Inconsistent API patterns
    - Missing error handling
    - Poor separation of concerns
  </Tab>
</Tabs>

**Architectural patterns checked**:
- API consistency (REST conventions, error handling)
- Authentication/Authorization patterns
- Data access patterns (repository, query optimization)
- Error handling consistency
- Security layers (validation, sanitization, rate limiting)
- Performance patterns (caching, lazy loading, code splitting)
- Integration patterns (events, messaging, webhooks)

## Project Structure Analysis

### WIS-15: New Feature Analysis

```bash
*analyze-project-structure
```

**Analyzes**:
1. **Codebase structure**: Folders, modules, organization
2. **Existing patterns**: API routes, services, components
3. **Dependencies**: Libraries, frameworks, tools
4. **Architecture style**: Monolith, microservices, modular
5. **Integration points**: APIs, databases, external services
6. **Testing setup**: Test frameworks, coverage

**Output**: Detailed report for implementing new features

```markdown
# Project Structure Analysis

## Architecture Style
Monolithic Node.js application with React frontend

## Key Patterns
- API: RESTful with Express.js
- Data: Repository pattern with Supabase
- Frontend: Component-based with Zustand state

## Integration Points
- Supabase (PostgreSQL + Auth + Storage)
- Stripe (Payments)
- SendGrid (Email)

## Recommendations for New Feature
1. Follow existing repository pattern
2. Add API route in `api/src/routes/`
3. Create Zustand store in `app/stores/`
4. Add tests in `tests/integration/`
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Architecture too complex" icon="circle-exclamation">
    **Signs**:
    - Too many layers
    - Over-engineering for current needs
    - Complex patterns for simple use cases
    
    **Solution**:
    1. Apply "boring technology" principle
    2. Start simple, add complexity as needed
    3. Focus on current requirements, not future-proofing
  </Accordion>
  
  <Accordion title="Database vs Architecture decision" icon="question">
    **Rule of thumb**:
    - **Aria**: "Should we use PostgreSQL or MongoDB?" (System-level)
    - **Dara**: "Design the users table schema" (Implementation-level)
    
    **When unsure**: Start with Aria, who will delegate if needed.
  </Accordion>
  
  <Accordion title="CodeRabbit architectural review taking long" icon="clock">
