# Brownfield Migration

Brownfield development refers to adding features, fixing bugs, or modernizing existing software projects. AIOX provides specialized workflows and tools for integrating with existing codebases safely and incrementally.

## What is Brownfield Development?

Unlike greenfield (new) projects, brownfield work requires:

- Understanding existing code and architecture
- Respecting existing constraints and patterns
- Ensuring new changes integrate seamlessly
- Avoiding breaking existing functionality
- Working with legacy systems and technical debt

## When to Use AIOX for Brownfield

AIOX is valuable for brownfield projects when you need to:

- Add significant new features to existing applications
- Modernize legacy codebases
- Integrate new technologies or services
- Refactor complex systems
- Fix bugs requiring architectural understanding
- Document undocumented systems

## When NOT to Use Brownfield Flow

If you just completed an MVP with AIOX and want to continue post-MVP development:

1. Talk to the PM agent to create a new epic in the PRD
2. Shard out the epic with the SM agent
3. Update architecture documents with the Architect agent
4. Continue with normal AIOX workflow

The full brownfield flow is for **integrating AIOX into existing non-AIOX projects**.

## Choose Your Approach

### Approach A: PRD-First (Recommended)

**Best for**: Large codebases, monorepos, or when you know exactly what to build

1. Create PRD first to define requirements
2. Document only relevant areas based on PRD needs
3. More efficient - avoids documenting unused code

**Advantages**:
- Focused documentation effort
- Clear scope from start
- Faster for large codebases
- Identifies required documentation upfront

### Approach B: Document-First

**Best for**: Smaller codebases, unknown systems, or exploratory changes

1. Document entire system first
2. Create PRD with full context
3. More thorough - captures everything

**Advantages**:
- Complete system understanding
- Discover hidden dependencies
- Better for exploration
- Comprehensive baseline documentation

## Complete Brownfield Workflow

### Phase 1: Discovery and Analysis

#### Step 1: Analyze Existing Project

Use the brownfield analyzer to understand the current state:

```javascript .aiox-core/development/tasks/analyze-brownfield.md
const { analyzeProject, formatMigrationReport } = require(
  './.aiox-core/infrastructure/scripts/documentation-integrity/brownfield-analyzer'
);

const targetDir = process.cwd();
const analysis = analyzeProject(targetDir);

// Display analysis report
const report = formatMigrationReport(analysis);
console.log(report);
```

**Analysis Output**:

```typescript
interface BrownfieldAnalysis {
  // Project flags
  hasExistingStructure: boolean;   // src/, lib/, tests/
  hasExistingWorkflows: boolean;   // CI/CD configs
  hasExistingStandards: boolean;   // Linting/formatting
  
  // Merge strategy
  mergeStrategy: 'parallel' | 'manual';
  
  // Detected tech stack
  techStack: string[];      // ['Node.js', 'TypeScript', 'Python']
  frameworks: string[];     // ['React', 'Express', 'Next.js']
  version: string | null;   // Project version
  
  // Configuration paths
  configs: {
    eslint: string | null;
    prettier: string | null;
    tsconfig: string | null;
    packageJson: string | null;
    githubWorkflows: string | null;
  };
  
  // Detected settings
  linting: string;      // 'ESLint' | 'Flake8' | 'none'
  formatting: string;   // 'Prettier' | 'Black' | 'none'
  testing: string;      // 'Jest' | 'Vitest' | 'pytest' | 'none'
  
  // Integration guidance
  recommendations: string[];
  conflicts: string[];
  manualReviewItems: string[];
  summary: string;
}
```

#### Step 2: Interpret Results

**Merge Strategy**:

| Strategy | Meaning | Action |
|----------|---------|--------|
| `parallel` | Safe to proceed with standard setup | Use standard AIOX setup |
| `manual` | Existing CI/CD requires review | Review workflows first |

**Review Manual Items**:

```bash
# Review GitHub workflows
ls -la .github/workflows/

# Check for conflicts with AIOX workflows
# Look for: quality-gate.yml, release.yml, staging.yml

# Review GitLab CI
cat .gitlab-ci.yml | grep -E "^[a-z]+:"

# Review CircleCI
cat .circleci/config.yml
```

**Handle Conflicts**:

```bash
# If docs/architecture/ exists:
# Option A: Rename existing
mv docs/architecture docs/legacy-architecture

# Option B: Configure alternate path
echo "AIOX_DOCS_PATH=docs/aiox-architecture" >> .env

# Create backup before integration
tar -czf backup-$(date +%Y%m%d).tar.gz .
```

### Phase 2: Documentation Strategy

#### PRD-First Approach (Recommended)

**Step 1: Create Brownfield PRD**

In Gemini Web (with codebase uploaded):

```bash
@pm
*create-doc brownfield-prd
```

The PM agent will:
- Ask about your enhancement requirements
- Explore the codebase to understand current state
- Identify affected areas needing documentation
- Create focused PRD with clear scope

**Key Advantage**: PRD identifies which parts of your codebase actually need documentation!

For example, if adding payment processing:
- Documents only: user service, API endpoints, database schemas, payment integrations
- Creates focused source tree showing payment-related code paths
- Skips: admin panels, reporting modules, unrelated microservices

**Step 2: Focused Documentation**

Still in Gemini Web, with PRD context:

```bash
@analyst
*document-project
```

The analyst will:
- Reference the PRD to understand scope
- Focus on relevant modules identified in PRD
- Skip unrelated areas to keep docs lean
- Generate ONE architecture document for all environments

#### Document-First Approach

**Best Approach - Gemini Web with 1M+ Context**:

1. Go to [Gemini Web](https://gemini.google.com)
2. Upload your project:
   - **Option A**: Paste GitHub repository URL
   - **Option B**: Upload up to 1000 files from src/
   - **Option C**: Zip project and upload archive
3. Load analyst agent: Upload `dist/agents/analyst.txt`
4. Run documentation: Type `*document-project`

The analyst generates comprehensive documentation of everything.

#### Option A: Full Brownfield Workflow (Major Changes)

**1. Create Brownfield PRD** (if not already done):

**Key Prompts You'll Encounter**:
- "What specific enhancement or feature do you want to add?"
- "Are there any existing systems this needs to integrate with?"
- "What are the critical constraints we must respect?"
- "What is your timeline and team size?"

**2. Create Brownfield Architecture**:

```bash
@architect
*create-doc brownfield-architecture
```

The architect will:
- Review the brownfield PRD
- Design integration strategy
- Plan migration approach if needed
- Identify technical risks
- Define compatibility requirements

**For Single Epic**:

```bash
@pm

```

Use when:
- Enhancement is well-defined and isolated

- Quick turnaround needed

**For Single Story**:

```bash

```

Use when:
- Bug fix or tiny feature
- Very isolated change
- No architectural impact
- Clear implementation path

## Brownfield Best Practices

### 1. Always Document First

Even if you think you know the codebase:

```bash
@analyst
*document-project
```

Benefits:
- Captures current state for AI agents
- Discovers undocumented patterns
- Reveals hidden dependencies
- Provides baseline for changes

### 2. Respect Existing Patterns

Brownfield templates specifically look for:

- Current coding conventions
- Existing architectural patterns
- Technology constraints
- Team preferences

**Example**: If project uses Prettier with 2-space indents, AIOX will generate code matching that style.

### 3. Plan for Gradual Rollout

Brownfield changes should include:

```javascript
// Feature flags for gradual rollout
if (featureFlags.newPaymentFlow) {
  return newPaymentService.process(order);
} else {
  return legacyPaymentService.process(order);
}

// Migration scripts
const migration = {
  up: async () => {
    // Add new columns
    await db.schema.table('users', (table) => {
      table.string('external_id').nullable();
    });
  },
  down: async () => {
    // Rollback changes
    await db.schema.table('users', (table) => {
      table.dropColumn('external_id');
    });
  },
};

// Backwards compatibility
const getUserData = (user) => {
  // Support both old and new formats
  return {
    id: user.id,
    name: user.fullName || user.name, // Fallback to old field
    email: user.email,
  };
};
```

### 4. Test Integration Thoroughly

Focus testing on:

```yaml
test-strategy:
  integration-points:
    - API endpoint compatibility
    - Database schema changes
    - External service calls
    - Event handling
  
  regression:
    - Existing user flows
    - Critical business logic
    - Edge cases
    - Error handling
  
  performance:
    - Response times
    - Database query efficiency
    - Memory usage
    - Concurrent requests
  
  data-migration:
    - Data integrity
    - Rollback capability
    - Zero downtime
    - Validation checks
```

### 5. Communicate Changes

Document in `docs/CHANGELOG.md`:

```markdown
## [2.0.0] - 2024-03-15

### Added
- New payment processing system with Stripe integration
- Support for subscription billing
- Payment method management UI

### Changed
- **BREAKING**: Payment API now uses `/api/v2/payments`
- Order schema includes `payment_method_id` field
- Checkout flow redesigned for better UX

### Deprecated
- `/api/v1/payments` endpoint (use v2, v1 supported until 2024-06-15)
- `PaymentService.processLegacy()` method

### Migration Guide
- Update API calls from `/api/v1/payments` to `/api/v2/payments`
- Run migration: `npm run migrate:payment-v2`
- Update payment webhooks to new format
- Test with Stripe test mode before production

### Rollback Plan
1. Set `ENABLE_PAYMENT_V2=false` in environment
2. Rollback database: `npm run migrate:rollback payment-v2`
3. Redeploy previous version
```

## Troubleshooting

### "The AI doesn't understand my codebase"

**Solution**: Re-run documentation with specific paths

```bash
@analyst
*document-project --paths "src/core,src/api,src/services"
```

### "Generated plans don't fit our patterns"

**Solution**: Update generated documentation with your conventions before planning

```markdown
## Coding Standards

Our project uses:
- 2-space indentation (not 4)
- Single quotes (not double)
- Functional components (not class components)
- Async/await (not promises)
- Jest for testing (not Vitest)
```

### "Too much boilerplate for small changes"

**Solution**: Use focused story creation

```bash
@pm
*brownfield-create-story
# Skip full PRD and architecture docs
```

### "Integration points unclear"

**Solution**: Provide more context during PRD creation

```
Integration requirements:
- Must use existing AuthService for authentication
- Database changes require migration scripts
- API endpoints follow /api/v2/:resource pattern
- Events published to RabbitMQ message bus
```
